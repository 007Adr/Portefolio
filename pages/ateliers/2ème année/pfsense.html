Contexte de l'atelier

Cet atelier de professionnalisation de 2ème année BTS SIO SISR consiste à déployer et configurer un pare-feu pfSense dans une infrastructure réseau virtualisée, puis à mettre en place un tunnel VPN OpenVPN permettant un accès distant sécurisé.

Le pfSense est équipé de deux interfaces réseau : une carte WAN reliée à la Livebox (réseau 192.168.1.0/24) et une carte LAN connectée au VNET étudiant (172.26.206.0/24). Un serveur Apache Linux est déployé derrière le pfSense et accessible depuis l'extérieur via redirection de port.

La seconde partie approfondit la sécurité avec la mise en place d'une PKI complète (Autorité de Certification, certificats serveur et utilisateur) et d'un serveur OpenVPN en mode accès distant, permettant de joindre le réseau interne depuis n'importe où de manière chiffrée.

1
Première Partie — Infrastructure pfSense & Apache
Configuration serveurs · NAT/PAT · Redirection de port
1
Installation pfSense avec 2 cartes réseaux
Déploiement depuis l'ISO netgate-installer. La carte WAN est configurée en DHCP sur le réseau 192.168.1.0/24 (VLAN992, reliée à la Livebox) avec réservation d'adresse statique. La carte LAN est fixée à 172.26.206.250 sur le VNET étudiant.
  • Ajout du hostname pfSense-206 et du domaine home.arpa dans System/General Setup
  • Deux interfaces réseau virtuelles : net0 (vlan992) et net1 (vlan206)
2
Déploiement du serveur Apache Linux
Installation d'une machine Linux sur le VNET étudiant (172.26.206.80). Le serveur accède à Internet via le pfSense qui assure le routage NAT. La page Apache affiche « Bonjour, je m'appelle Adrien Blaize / Je suis en BTS SIO ».
  • Interface réseau enp0s18 configurée avec l'IP statique 172.26.206.80/24
  • Passerelle par défaut : 172.26.206.250 (pfSense LAN)
3
Redirection de port NAT/PAT (Port Forwarding)
Configuration d'une règle NAT dans Firewall → NAT → Port Forward pour rediriger le trafic TCP entrant sur le port 26206 (WAN) vers le serveur Apache sur le port 80 (HTTP), à l'adresse interne 172.26.206.80.
  • Interface WAN · Protocole TCP · Port destination 26206 → 172.26.206.80:80
  • Réservation de l'IP 192.168.1.225 sur la Livebox via l'adresse MAC du pfSense
  • Redirection configurée sur la box par les professeurs (port 26206 mappé sur pfSense-206)

2
Deuxième Partie — VPN OpenVPN avec PKI complète
CA · Certificats · OpenVPN Server · Règles pare-feu · Tests
1
Créer l'Autorité de Certification (CA)
Dans System → Certificates → Authorities, création d'une CA interne auto-signée nommée CA-BLAIZE. Common Name : domblaize.local, Organisation : BLAIZE & Co, localisation France / Rhône-Alpes / Lyon. Validité : 10 ans.
2
Créer le certificat serveur
Génération d'un certificat interne Certificat-OpenVPN-AB signé par CA-BLAIZE, de type Server Certificate. Chiffrement RSA 2048 bits, algorithme SHA256, durée de validité 3650 jours. Common Name : Firewall-Pfsense-AB.
3
Créer un utilisateur et son certificat
Dans System → User Manager, création du compte adrien.blaize avec mot de passe sécurisé et génération simultanée du certificat utilisateur Certificat-VPN-Adrien, rattaché à CA-BLAIZE (RSA 2048 / SHA256 / 3650 jours).
4
Configurer le serveur OpenVPN
Dans VPN → OpenVPN → Servers, création du serveur « Accès distant OpenVPN AB » en mode SSL/TLS + User Auth (double authentification : certificat + identifiants).
  • Protocole UDP sur l'interface WAN · Port personnalisé 20694
  • CA : CA-BLAIZE · Certificat serveur : Certificat-OpenVPN-AB
  • Chiffrement : AES-256-GCM, CHACHA20-POLY1305 · DH 2048 bits · Auth SHA256
  • Tunnel IPv4 : 10.10.206.0/24 · Réseau local : 172.26.206.0/24
  • Connexions simultanées : 2 · DNS domain : domblaize.local · auth-nocache
5
Exporter la configuration OpenVPN
Installation du paquet openvpn-client-export via System → Package Manager. Génération automatique du fichier .ovpn et de l'installeur Windows pour le compte d'adrien.blaize. Le fichier pointe vers 90.65.104.37:20694.
6
Créer les règles de pare-feu
Quatre règles créées pour autoriser le trafic VPN et les flux internes :
  • WAN → UDP 20694 : « Accès Distant OpenVPN » — permet aux clients de s'authentifier
  • OpenVPN → ICMP → 172.26.206.0/24 : ping vers tout le LAN pour tester la connectivité
  • OpenVPN → TCP:22 → 172.26.206.80 : accès SSH au serveur Apache
  • OpenVPN → TCP/UDP:80 → 172.26.206.80 : accès HTTP au serveur web
  • OpenVPN → TCP:3389 → 172.26.206.80 : « Autoriser RDP via VPN » — bureau à distance
7
Tester et valider le tunnel VPN
Connexion réussie depuis le poste client Windows — adresse IP VPN attribuée : 10.10.206.6. La ligne Initialization Sequence Completed confirme le tunnel opérationnel.
  • Ping ✅ — ICMP vers 172.26.206.80 : 4 paquets envoyés, 0 perdu, latence 2-4ms
  • SSH ✅ — Connexion PuTTY vers 172.26.206.80:22 validée via le tunnel
  • HTTP ✅ — Page Apache accessible depuis le navigateur via IP interne
  • RDP ✅ — Bureau à distance xRDP opérationnel (1.01 MiB de trafic constaté)
  • Wireshark ✅ — Paquets UDP chiffrés port 20694 visibles sur l'interface physique
