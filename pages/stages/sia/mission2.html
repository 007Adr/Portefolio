SP2 SIA — Déploiement d'un Serveur de Backup Automatisé
Proxmox · VM Ubuntu · Crontab · Rclone · Discord Webhook · GitLab
Contexte

Dans le cadre de mon stage au SIA BDE INSA Lyon, j'ai été chargé de déployer et configurer un serveur de backup automatisé destiné à sauvegarder les bases de données critiques de l'infrastructure (PostgreSQL et MariaDB).

L'objectif était de mettre en place une solution fiable, automatique et observable : les sauvegardes devaient se déclencher la nuit via crontab, copier les données depuis le VPS de production, appliquer une politique de rétention, et notifier l'équipe en temps réel sur Discord via un webhook.

Ce projet m'a permis de travailler de bout en bout sur la création d'une VM sous Proxmox, la configuration des disques, la migration des scripts existants depuis GitLab, et l'automatisation complète du processus de backup.

Environnement technique
  • Hyperviseur : Proxmox VE — VM ID 100 (ubuntu-backup)
  • Système : Ubuntu Server 24.04.3 LTS
  • Disques : 2 disques — sda (32G OS) · sdb (1.8T Backups)
  • Outils : Rclone, fdisk, mkfs.ext4, crontab, Bash
  • Sources de backup : VPS infra.bde-insa-lyon.fr — PostgreSQL + MariaDB + Duplicati
  • Notification : Discord Webhook via variable d'environnement système
  • Versioning : GitLab SIA — dépôt privé infra, deploy token infra-mde

1 — Configuration des disques

La première étape consiste à stopper la VM depuis le shell Proxmox, puis à identifier les disques disponibles avec lsblk. Cela permet de visualiser l'ensemble des disques et leurs partitions avant toute modification.

qm stop 100 pour stopper la VM · lsblk pour lister les disques

qm set 100 -scsi1 /dev/sdb — rattachement du disque 1.8T à la VM

La commande qm set 100 -scsi1 /dev/sdb permet de choisir le type de disque et de le rattacher à la VM. On peut ensuite vérifier via qm config 100 | grep scsi que le disque est bien associé, puis redémarrer la VM.

Vérification de la config SCSI et redémarrage de la VM

Une fois la VM redémarrée, je prépare le disque /dev/sdb proprement en supprimant l'ancienne partition, en créant une nouvelle et en formatant en ext4 :

# Partitionner le disque avec fdisk
sudo fdisk /dev/sdb
  d     # Supprimer partition existante
  1     # Numéro sdb1
  n     # Nouvelle partition
  # Entrées (partition 1, début défaut, fin 100%)
  w     # Écrire la table et quitter

# Formater en ext4
sudo mkfs.ext4 -F /dev/sdb1

# Monter le disque
sudo mount /dev/sdb1 /mnt

# Vérifier
sudo fdisk -l /dev/sdb

Structure des disques dans la VM — sdb1 monté sur /mnt

2 — Migration des données & configuration des scripts

Pour récupérer les scripts existants, j'ai eu accès au dépôt Git privé côté infra. Cependant, le mot de passe du token d'accès n'était plus connu. Mon maître de stage a géré cette étape en regénérant un deploy token GitLab (infra-mde) permettant un accès en lecture seule au dépôt.

Deploy token infra-mde — accès read_repository sur le dépôt infra GitLab

Une fois les scripts récupérés, j'ai ajouté les variables d'environnement nécessaires à leur fonctionnement. Le script pull_backups.sh utilise notamment rclone pour copier les backups depuis le VPS vers le serveur local, avec une politique de rétention de 30 jours.

Configuration du script pull_backups.sh — rétention 30j, VPS source, chemins locaux

J'ai également ajouté une variable d'environnement DISCORD_WEBHOOK_URL dans le fichier /etc/environment, exportée globalement sur le système. Cette variable est utilisée par les scripts pour envoyer des notifications Discord à chaque exécution.

DISCORD_WEBHOOK_URL déclarée dans /etc/environment

💬 Webhook Discord : Les webhooks sont une manière simple de poster des messages dans Discord depuis des applications externes via une URL HTTP. À chaque lancement manuel ou automatique des scripts, un message est envoyé sur le canal Discord de l'équipe SIA avec le statut détaillé de chaque copie.

3 — Automatisation avec Crontab

Les backups sont automatisés grâce à crontab. La tâche planifiée exécute le script pull_backups.sh tous les jours à 03h44, en redirigeant les sorties (stdout et stderr) dans un fichier de log dédié.

Règle crontab — exécution quotidienne à 03h44

# Syntaxe : minute heure jour mois jour-semaine commande
44 3 * * * /home/sia/scripts/pull_backups.sh >> /mnt/backups/logs/pull_backups.log 2>&1

# "44 3 * * *" = à 03h44 tous les jours
# ">> fichier.log 2>&1" = stdout + stderr dans le même fichier de log
⏱️ Note : Bien que la tâche soit planifiée à 03h44, le script prend du temps à s'exécuter (copies réseau, vérifications). En pratique, les notifications Discord arrivent aux alentours de 04h00, ce qui correspond au temps de traitement complet du Backup Puller.

Statut succès — toutes les copies réussies ✅

4 — Scripts & Documentation

L'ensemble des scripts est versionné sur le dépôt GitLab privé du SIA. Le projet contient 4 fichiers : les scripts de production actuels et leurs versions de sauvegarde (_save).

MdE scripts/
├── pull_backups.sh       # Script pull — version production
├── send_backup.sh        # Script send — version production
├── pull_backups_save.sh  # Script pull — ancienne génération
├── send_backup_save.sh   # Script send — ancienne génération
└── README.md             # Documentation complète

pull_backups.sh — Copie des backups depuis le VPS

Ce script exécute une pipeline complète qui :

  • Copie les backups PostgreSQL dans un dossier commun
  • Copie les backups MariaDB dans un dossier commun
  • Copie les backups Duplicati dans un dossier organisé par date
  • Applique une politique de rétention (30 jours)
  • Écrit les backups dans /home/sia/backups et génère des logs détaillés

send_backup.sh — Envoi sélectif vers le serveur OVH

Ce script permet de sélectionner un backup spécifique (PostgreSQL, MariaDB ou Duplicati) à une date précise et de l'envoyer dans une dropzone sur le serveur principal OVH.

Le script attend un format de nommage strict :

# Backups de BDD — format attendu
backups/postgresql-backups/nombase-YYYYMMDD.sql
backups/mariadb-backups/nombase-YYYYMMDD.dump

# Backups Duplicati — format attendu
backups/duplicati/YYYY-MM-DD/

⚙️ Installation & configuration

🔑 SSH

  • Générer une paire de clés ed25519
  • Créer un user backup_puller sur le VPS source
  • Droits lecture/écriture uniquement sur les dossiers nécessaires
  • Ajouter la clé publique dans authorized_keys

☁️ Rclone

  • Configurer via rclone config
  • Créer un remote nommé sia
  • Host : infra.bde-insa-lyon.fr
  • Renseigner les clés SSH privée et publique
🛡️ Bonnes pratiques : Ne pas oublier de configurer les mises à jour automatiques (unattended-upgrades), un firewall (ufw) et Twingate VPN pour sécuriser l'accès au serveur de backup.
